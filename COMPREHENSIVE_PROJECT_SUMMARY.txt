SECTION 1: PROJECT OVERVIEW & PURPOSE
================================================================================

PROJECT NAME:
  MobileGymTrack (Trackd) - Fitness Tracking PWA

VISION:
  Deliver a mobile-first, offline-capable fitness tracking application that 
  enables gym enthusiasts to plan, execute, and analyze their strength training 
  workouts with professional-grade analytics and customizable training splits.

PRIMARY USE CASES:
  1. Workout Planning: Create custom workout templates with exercise-specific
     parameters (sets, reps, target weight)
  2. Workout Execution: Real-time workout tracking with exercise progress,
     set completion, and rest timer management
  3. Performance Analytics: Comprehensive personal records, 1RM progression,
     lifetime volume tracking, and strength rankings
  4. Workout History: Complete session history with volume calculations,
     completion metrics, and heatmap activity visualization
  5. Customization: Three themed UI options, language support (Polish/English),
     user settings persistence

TARGET USERS:
  - Intermediate to advanced fitness enthusiasts
  - Strength athletes tracking periodized training
  - Gym-goers requiring detailed workout documentation

KEY FEATURES:
  ✓ Offline-capable PWA (installable on mobile devices)
  ✓ Real-time data synchronization with Supabase backend
  ✓ Session recovery (resume workouts after app crash/interruption)
  ✓ Multiple themed UI variants (Classic, Professional, Metal)
  ✓ Bilingual support (English, Polish)
  ✓ Advanced statistics (1RM calculations, volume tracking, rankings)
  ✓ Personal records management and progression tracking
  ✓ Activity heatmap for monthly workout visualization
  ✓ Responsive design for mobile and tablet devices

SECTION 3: DATABASE ARCHITECTURE & SCHEMA
================================================================================

DATABASE PLATFORM:
  - Supabase (PostgreSQL)
  - Real-time capabilities enabled
  - Row-Level Security (RLS) enforced on all tables
  - Automatic timestamp management (created_at, updated_at)

CORE TABLES:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TABLE: auth.users (Supabase Auth)
────────────────────────────────────
  Provided by Supabase Authentication
  - id (UUID, Primary Key)
  - email (Unique)
  - encrypted_password
  - confirmed_at (Email verification timestamp)
  
  Used for: User identity, email authentication, session management

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TABLE: workout_splits
──────────────────────
  Structure:
    - id (BIGINT, Primary Key, Auto-increment)
    - user_id (UUID, Foreign Key → auth.users)
    - name (TEXT, Not Null)
    - created_at (TIMESTAMP, Default: UTC now)
    - updated_at (TIMESTAMP, Default: UTC now)

  Purpose:
    Organizing multiple workout templates into named splits
    (e.g., "Push/Pull/Legs", "Upper/Lower", "Full Body")

  RLS Policies:
    • SELECT: User can view only own splits (auth.uid() = user_id)
    • INSERT: User can create splits
    • UPDATE: User can update own splits
    • DELETE: User can delete own splits

  Relationships:
    → workout_templates (1:Many, split_id foreign key, SET NULL on delete)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TABLE: workout_templates
────────────────────────
  Structure:
    - id (BIGINT, Primary Key, Auto-increment)
    - user_id (UUID, Foreign Key → auth.users)
    - split_id (BIGINT, Foreign Key → workout_splits, Nullable)
    - name (TEXT, Not Null) - Workout template name
    - description (TEXT, Nullable)
    - exercises (JSONB, Default: [])
    - created_at (TIMESTAMP, Default: UTC now)
    - updated_at (TIMESTAMP, Default: UTC now)

  Exercise JSON Structure:
    [
      {
        "name": "Bench Press",
        "sets": "4",
        "reps": "8-10",
        "weight": "100"
      },
      ...
    ]

  Purpose:
    Reusable workout templates that users execute multiple times

  RLS Policies:
    • SELECT: User can view only own templates
    • INSERT: User can create templates
    • UPDATE: User can update own templates
    • DELETE: User can delete own templates

  Relationships:
    ← workout_splits (Many:1, split_id, SET NULL on delete)
    → completed_sessions (1:Many, workout_id foreign key, CASCADE on delete)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TABLE: completed_sessions
─────────────────────────
  Structure:
    - id (BIGINT, Primary Key, Auto-increment)
    - user_id (UUID, Foreign Key → auth.users)
    - workout_id (BIGINT, Foreign Key → workout_templates)
    - completed_at (TIMESTAMP, Default: UTC now)
    - exercises (JSONB, Default: [])
    - duration (INTEGER, Default: 0, in seconds)
    - created_at (TIMESTAMP, Default: UTC now)

  Exercise JSON Structure (with metadata):
    {
      "name": "Bench Press",
      "data": [
        {
          "exercise": "Bench Press",
          "sets": [
            {"completed": true, "reps": "10", "weight": "100"},
            {"completed": true, "reps": "8", "weight": "105"},
            ...
          ]
        }
      ]
    }

  Purpose:
    Historical record of all completed workout sessions
    Stores actual performance data (weight, reps) vs. planned targets

  RLS Policies:
    • SELECT: User can view only own sessions
    • INSERT: User can create sessions
    • UPDATE: User can update own sessions
    • DELETE: User can delete own sessions

  Relationships:
    ← workout_templates (Many:1, workout_id)
    → personal_records (implicit, used for 1RM calculations)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TABLE: personal_records
───────────────────────
  Structure:
    - id (BIGINT, Primary Key, Auto-increment)
    - user_id (UUID, Foreign Key → auth.users)
    - exercise (TEXT, Not Null)
    - weight (NUMERIC, Default: 0)
    - reps (NUMERIC, Default: 0)
    - created_at (TIMESTAMP, Default: UTC now)
    - updated_at (TIMESTAMP, Default: UTC now)

  Purpose:
    Tracks personal records (max weight achieved) for each exercise
    Updated after every workout session based on actual performance

  RLS Policies:
    • SELECT: User can view only own records
    • INSERT: User can create records
    • UPDATE: User can update own records
    • DELETE: User can delete own records

  Relationships:
    ← completed_sessions (implicit relationship via exercise matching)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TABLE: user_settings
────────────────────
  Structure:
    - id (BIGINT, Primary Key, Auto-increment)
    - user_id (UUID, Unique, Foreign Key → auth.users)
    - settings (JSONB, Default: {"language": "en"})
    - theme (TEXT, Default: 'classic')
    - created_at (TIMESTAMP, Default: UTC now)
    - updated_at (TIMESTAMP, Default: UTC now)

  Purpose:
    Stores user preferences and settings
    One record per user (UNIQUE user_id constraint)

  Settings JSON Example:
    {
      "language": "en",
      "units": "kg",
      "notifications": true
    }

  Theme Options:
    - "classic" (Slate Blue with Orange accent)
    - "professional" (Pure Black with Cyan accent)
    - "metal" (Pure Black with Red accent)

  RLS Policies:
    • SELECT: User can view only own settings
    • INSERT: User can create settings
    • UPDATE: User can update own settings
    • DELETE: User can delete own settings

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DATA FLOW DIAGRAM:

  User (auth.users)
    ├─→ workout_splits (1:Many)
    │    └─→ workout_templates (1:Many)
    │         └─→ completed_sessions (1:Many)
    │              └─→ [exercises JSONB with performance data]
    │
    ├─→ workout_templates (1:Many) [ungrouped]
    │    └─→ completed_sessions (1:Many)
    │
    ├─→ completed_sessions (1:Many)
    │    └─→ [exercises JSONB with actual performance]
    │
    ├─→ personal_records (1:Many)
    │    └─→ [exercise names with max weight achieved]
    │
    └─→ user_settings (1:1)
         └─→ theme + language preferences

SECURITY MODEL:
  - Row-Level Security (RLS) enabled on all tables
  - All queries filtered by auth.uid() = user_id
  - Users can only access their own data
  - Cascading deletes prevent orphaned records
  - Foreign key constraints maintain referential integrity

CORE CONTEXTS & PROVIDERS:

1. AuthContext (src/AuthProvider.jsx)
   ─────────────────────────────────
   Provides:
     - user: Current authenticated user (Supabase user object)
     - logout(): Sign out user
     - error: Error state during auth operations
   
   Features:
     - Automatic session detection on mount
     - Real-time auth state subscription
     - Silent user refresh on app wake
   
   Usage:
     const { user, logout } = useAuth();

2. ThemeContext (src/ThemeContext.jsx)
   ──────────────────────────────────
   Provides:
     - theme: Current theme ID ('classic', 'professional', 'metal')
     - switchTheme(newTheme): Change theme (updates database)
     - availableThemes: Array of available themes
     - themeInfo: Color definitions for current theme
   
   Features:
     - Persists theme selection to user_settings table
     - Applies theme-specific CSS class to <html>
     - Optimistic UI updates
     - Theme definitions with color palette
   
   Theme Colors:
     Classic:
       - Background: #0f172a (Slate Blue)
       - Accent: #f97316 (Orange)
     Professional:
       - Background: #000000 (Pure Black)
       - Accent: #00e5ff (Cyan)
     Metal:
       - Background: #000000 (Pure Black)
       - Accent: #ff0000 (Red)

3. ToastContext (src/ToastContext.jsx)
   ──────────────────────────────────
   Provides:
     - success(message): Show success notification
     - error(message): Show error notification
     - info(message): Show info notification
   
   Features:
     - Auto-dismiss after 3-5 seconds
     - Multiple toasts can queue
     - Theme-aware styling

SECTION 6: STATE MANAGEMENT & DATA FLOW
================================================================================

STATE ARCHITECTURE:

The application uses a combination of:
  1. React Context (global state)
     - AuthContext: Current user
     - ThemeContext: UI theme
     - ToastContext: Notifications
  
  2. Component State (local state)
     - Individual useState hooks
     - useCallback for memoized functions
     - useEffect for side effects
  
  3. Browser Storage (persistence)
     - localStorage for active session
     - Theme preference in user_settings table
     - Supabase for authoritative data

KEY DATA FLOWS:

1. USER AUTHENTICATION
   ─────────────────────
   Step 1: User navigates to app
   Step 2: AuthProvider checks Supabase session
   Step 3: If authenticated → Show authenticated app
           If not → Show Auth component
   Step 4: Auth component submits email/password
   Step 5: Supabase creates user in auth.users
   Step 6: System creates user_settings record
   Step 7: App fetches completed_sessions and personal_records
   Step 8: ThemeContext loads theme preference
   Step 9: Home page displays

2. WORKOUT CREATION
   ─────────────────
   User → CreateWorkout form
     ↓
   Input: Workout name, exercises (name, sets, reps, weight)
     ↓
   Validation: Name required, at least 1 exercise
     ↓
   Save: Supabase INSERT to workout_templates
     ↓
   JSON stored: exercises array with all data
     ↓
   Refresh: Templates list updates in Home
     ↓
   User can now execute this template

3. WORKOUT EXECUTION
   ──────────────────
   User clicks "Start Workout" on template
     ↓
   WorkoutPlayer initializes state from template
     ↓
   FOR EACH EXERCISE:
     • User enters weight/reps for each set
     • localStorage auto-saves progress
     • Rest timer runs between sets
     • User marks sets as complete
   ↓
   User clicks "Finish Workout"
     ↓
   Validation: At least one set completed
     ↓
   Prepare data:
     • Wrap exercises with metadata {name, data}
     • Calculate duration
     • Get completed_at timestamp
   ↓
   Save to Supabase:
     INSERT INTO completed_sessions:
       - user_id
       - workout_id
       - exercises (JSONB)
       - duration
       - completed_at
   ↓
   Update personal records:
     FOR EACH EXERCISE:
       - Calculate 1RM from sets
       - Compare with existing PR
       - UPDATE personal_records if new max
   ↓
   Show success message with volume
     ↓
   Clear localStorage
     ↓
   Home page refreshes stats

4. SESSION RECOVERY
   ──────────────
   App crashes mid-workout
     ↓
   User reopens app
     ↓
   AuthProvider re-authenticates
     ↓
   App reads localStorage (STORAGE_KEY)
     ↓
   IF valid session data exists:
     • Set as recoveredSession prop
     • Pass to Home component
     • Home initializes WorkoutPlayer with recovered data
   ↓
   User can continue from where they left off

5. STATISTICS & ANALYTICS
   ──────────────────────
   Home component calculates:
     • totalLifetimeVolume = SUM(exercises.data[].sets[].weight × reps)
     • totalSessions = COUNT(completed_sessions)
     • averagePerSession = totalLifetimeVolume / totalSessions
     • currentRank = Tier based on volume
   ↓
   Display with animations:
     • useCountUpAnimation for smooth number transitions
     • AnimatePresence for entrance/exit effects
   ↓
   On session delete:
     • Recalculate metrics
     • Animate count-down
     • Update heatmap
